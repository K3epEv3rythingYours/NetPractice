### The Human Frustration

**The Year:** 1844  
**The Problem:** Messages took WEEKS to cross the Atlantic by ship  
**The Hero:** Samuel Morse â€” not a scientist, but a portrait painter

### The Breakthrough Moment

Earlier telegraph designs proposed **26 separate wires** â€” one for each letter of the alphabet. Imagine the mess!

Morse had a different insight: **A single wire could carry all information by varying the TIMING of electrical pulses.**

```
   SHORT PULSE (dot)  = .
   LONG PULSE (dash)  = -
   
   "S" = ...
   "O" = ---
   "S" = ...
   
   "SOS" = ... --- ...
```

**May 24, 1844:** Morse transmitted "What hath God wrought?" from Washington D.C. to Baltimore.

### The Core Insight

> **Information is abstract, not physical.**  
> It can be represented by ANY two distinguishable states.  
> Dots/dashes. Open/closed. High/low voltage. 1/0.  
> This abstraction makes digital communication universal.

## Chapter 6: The Visionaries Who Dreamed It

### J.C.R. Licklider: The Prophet

**"Lick"** (1915-1990) provided the vision. While at MIT in 1957, he discovered that **85% of his time was spent preparing to think** â€” gathering data, making graphs â€” and only 15% actually thinking.

His insight: **Computers should be partners to human thought, not just calculators.**

In April 1963, Licklider wrote a memo addressed to:

> **"Members and Affiliates of the Intergalactic Computer Network"**

His vision: an *"electronic commons open to all."*

### Leonard Kleinrock: The Mathematician

Kleinrock's 1961 PhD thesis developed the theory of **packet switching** â€” breaking messages into small pieces that travel independently and reassemble at the destination.

Revolutionary insight: Instead of dedicating an entire phone line to one conversation (circuit switching), you could **share bandwidth** among many communications.

## Chapter 32: Static vs Dynamic Routing

### Static Routing

**Manually configured routes** that don't change unless admin modifies them.

| Pros | Cons |
|------|------|
| Simple | Doesn't adapt to failures |
| Predictable | Manual effort at scale |
| No protocol overhead | No automatic failover |
| Secure | Error-prone for large networks |

**Use when:** Home networks, small offices, single exit point

### Dynamic Routing

**Routes learned automatically** through protocols (OSPF, BGP, RIP).

| Pros | Cons |
|------|------|
| Auto-adapts to changes | More complex |
| Scales well | Uses CPU/memory |
| Automatic failover | Convergence time after changes |
| Less manual work | Security considerations |

**Use when:** Enterprise networks, ISPs, redundant paths

**For NetPractice:** All routes are static (manual configuration).

## Level 1: Basic Connectivity

### Concept Tested
Devices on the same network need matching network portions.

### Strategy
1. Find the **fixed IP** (the one you can't change)
2. Determine the **network range** from that IP + mask
3. Pick **valid unused IPs** within that range
4. Avoid network ID and broadcast address

### Example
```
Fixed: 192.168.1.10/24
Network: 192.168.1.0
Usable range: 192.168.1.1 - 192.168.1.254
Broadcast (don't use): 192.168.1.255

Valid choices for other device: 192.168.1.11, .12, .20, etc.
```

## Level 2: Subnet Masks

### Concept Tested
All devices on same LAN MUST have identical subnet masks.

### Strategy
1. Check if any masks are **fixed**
2. All devices on same network segment need **SAME mask**
3. Calculate valid IP ranges based on that mask
